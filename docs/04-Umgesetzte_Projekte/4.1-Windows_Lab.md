# 🧪 Umgesetzte Lab-Projekte

> Dieses Projekt simuliert eine vollständige Windows-Unternehmensumgebung — aufgebaut als virtuelle Maschinen auf meinem Proxmox-Homelab (Node 2 Deutschland).  
> Ziel: Praxisnahe Umsetzung realer Enterprise-IT mit Active Directory, Netzwerkdiensten und zentraler Sicherheitsverwaltung.

---

## 🖥️ Technische Basis

Das gesamte Projekt läuft auf echter Hardware in meinem Homelab — nicht in einer emulierten Umgebung auf einem Laptop.

| Eigenschaft         | Details                                                 |
| ------------------- | ------------------------------------------------------- |
| **Hypervisor**      | Proxmox VE                                              |
| **Host**            | Node 2 Deutschland — Selbst gebauter Tower (32 GB DDR4) |
| **Rolle des Nodes** | On-Demand Testing & Development                         |
| **VM-Plattform**    | KVM/QEMU via Proxmox                                    |

Alle virtuellen Maschinen (DC01, PC01, PC02) laufen als eigenständige VMs auf diesem Node und kommunizieren über ein isoliertes internes Netzwerk.

## 🏗️ 01 — Infrastruktur & VM-Setup

**Ziel:** Aufbau einer vollständig virtualisierten Windows-Unternehmensumgebung auf Proxmox als sichere, isolierte Testumgebung.

**VM-Übersicht:**

| VM   | Rolle                             | OS                  | IP                         |
| ---- | --------------------------------- | ------------------- | -------------------------- |
| DC01 | Domain Controller (AD, DNS, DHCP) | Windows Server 2022 | `192.168.10.10` (statisch) |
| PC01 | Domänenclient                     | Windows 11 Pro      | DHCP                       |
| PC02 | Domänenclient                     | Windows 11 Pro      | DHCP                       |

**Was ich umgesetzt habe:**

- Alle VMs in Proxmox erstellt und konfiguriert (CPU, RAM, Disk, Netzwerk)
- Hybrides Netzwerkmodell: internes VM-Netzwerk (`labnet`, `192.168.10.0/24`) + NAT für Internetzugang via DC01
- DC01 als zentraler Knotenpunkt mit statischer IP und Routing-Funktion
- Domain Join aller Clients in die Domäne `lab.local`

**Verifikation:**

| Test                  | Befehl                      | Ergebnis        |
| --------------------- | --------------------------- | --------------- |
| Netzwerkkonnektivität | `ping DC01`                 | ✅ Erreichbar   |
| DNS-Auflösung         | `nslookup DC01`             | ✅ Korrekte IP  |
| Domain Controller     | `nltest /dsgetdc:lab.local` | ✅ DC01 erkannt |
| Client-Kommunikation  | `ping PC02`                 | ✅ Erreichbar   |

**Ergebnis:** Stabile VM-Umgebung auf Proxmox — alle Hosts kommunizieren fehlerfrei, der Domain Controller ist vollständig erreichbar.

## 🌐 02 — Netzwerk & Dienste

**Ziel:** Strukturierte Client-Server-Kommunikation mit zentralen Netzwerkdiensten (DNS & DHCP) über den Domain Controller.

**Netzwerktopologie:**

```
[Internet]
     │
┌────┴────┐
│  DC01   │  NIC1: NAT (Proxmox → Internet)
│         │  NIC2: Internes Netzwerk (labnet)
└────┬────┘
     │
  192.168.10.0/24
  ┌──┴───────┬──────────┐
DC01       PC01        PC02
.10        DHCP        DHCP
```

**DNS:**

| Parameter | Wert                                |
| --------- | ----------------------------------- |
| Zone      | `lab.local`                         |
| Server-IP | `192.168.10.10`                     |
| Forwarder | Externer DNS (für Internetanfragen) |

**DHCP:**

| Parameter        | Wert                                |
| ---------------- | ----------------------------------- |
| Adressbereich    | `192.168.10.100` – `192.168.10.200` |
| Standard-Gateway | `192.168.10.10` (DC01)              |
| DNS-Option       | `192.168.10.10`                     |
| Leasedauer       | 8 Tage                              |

**Verifiziert mit:** `ipconfig /all`, `nslookup dc01.lab.local`, `echo %logonserver%`

**Ergebnis:** Clients beziehen automatisch ihre vollständige Netzwerkkonfiguration — Namensauflösung und Kommunikation laufen stabil.

## 🖥️ 03 — Active Directory

**Ziel:** Zentrale Identitätsverwaltung als Grundlage für GPOs, Single Sign-On und rollenbasierte Zugriffskontrolle.

**OU-Struktur:**

```
lab.local
├── OU=Users
│   ├── IT
│   └── HR
├── OU=Computers
│   ├── Clients
│   └── Servers
└── OU=Groups
```

**Benutzer:**

| Benutzername   | Name              | OU       | Gruppe |
| -------------- | ----------------- | -------- | ------ |
| `t.mustermann` | Thomas Mustermann | Users/IT | GG_IT  |
| `a.schmidt`    | Anna Schmidt      | Users/HR | GG_HR  |

**Gruppen:**

| Gruppe | Typ               | Zweck                              |
| ------ | ----------------- | ---------------------------------- |
| GG_IT  | Sicherheitsgruppe | Zugriffssteuerung für IT-Abteilung |
| GG_HR  | Sicherheitsgruppe | Zugriffssteuerung für HR-Abteilung |

**Was ich umgesetzt habe:**

- OU-Struktur nach Funktion — orientiert an echten Enterprise-Umgebungen
- Zentrale Benutzerverwaltung: keine lokalen Konten auf den Clients
- Gruppenmitgliedschaften für granulare Rechtevergabe vorbereitet
- Domänen-Login auf PC01 und PC02 erfolgreich getestet

**Ergebnis:** Skalierbare AD-Struktur — neue Benutzer, Abteilungen oder Richtlinien lassen sich jederzeit sauber integrieren.

## ⚙️ 04 — Gruppenrichtlinien (GPO)

**Ziel:** Zentrale, automatisierte Absicherung aller Clients über Gruppenrichtlinien — ohne manuelle Konfiguration an jedem Gerät.

**GPO-Übersicht:**

| GPO                 | Verknüpft mit         | Zweck                                       |
| ------------------- | --------------------- | ------------------------------------------- |
| Password Policy     | Default Domain Policy | Passwort-Sicherheit domänenweit erzwingen   |
| Security Baseline   | OU=Computers          | Systemzugriff einschränken                  |
| Client Restrictions | OU=Clients            | Einheitliche, kontrollierte Client-Umgebung |

**GPO 1 — Passwort-Richtlinie:**

| Einstellung      | Wert        |
| ---------------- | ----------- |
| Komplexität      | Aktiviert   |
| Mindestlänge     | 12 Zeichen  |
| Passwort-Verlauf | 10 Einträge |
| Maximales Alter  | 90 Tage     |
| Minimales Alter  | 1 Tag       |

> ⚠️ Als **Domain Password Policy** umgesetzt (Default Domain Policy) — Passwort-Richtlinien müssen domänenweit greifen und funktionieren nicht korrekt über eine normale OU-GPO.

**GPO 2 — Security Baseline:**

- Zugriff auf Systemsteuerung gesperrt
- CMD deaktiviert
- Task-Manager eingeschränkt

**GPO 3 — Client Restrictions:**

- Desktop-Einstellungen eingeschränkt
- USB-Nutzung deaktiviert
- Softwareinstallation gesperrt

**Anwendung & Verifikation:**

```cmd
gpupdate /force   # Richtlinien sofort auf den Client anwenden
gpresult /r       # Alle angewendeten GPOs anzeigen und prüfen
```

**Ergebnis:** Alle Richtlinien greifen nachweislich im Alltag — Systemsteuerung blockiert, CMD gesperrt, USB eingeschränkt. Zentrale, einheitliche Verwaltung der gesamten Domäne ohne manuelle Eingriffe an den Clients.
